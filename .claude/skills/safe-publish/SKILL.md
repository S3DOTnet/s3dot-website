# Safe Publish

## 目的

コミット・Push前の安全確認を徹底し、検証済みの意図した変更だけをGitで管理する。Pushはユーザーから明示的な「Pushしてください」がある場合に限る。

## コミットまでの手順（順番厳守）

1. `git status` で変更・未追跡ファイルを確認する
2. `git diff` と `git diff --staged` で差分内容を確認する
3. `npm run lint` を実行し、終了状態とエラー・警告を確認する
4. `npm run build` を実行し、ビルドの成功を確認する
5. 検証失敗、秘密情報の疑い、想定外の変更がある場合は、ステージ・コミットせずに停止して日本語で報告する
6. 問題がなければ、意図した変更ファイルだけをステージする
7. ステージ済み差分を再確認し、簡潔な日本語のコミットメッセージでコミットする
8. 変更対象、検証結果、コミットID・コミットメッセージを日本語で報告し、Pushせずに停止する

## Pushを明示された場合の手順

1. ユーザーが明示的に「Pushしてください」と指示したことを確認する
2. `git status`、現在のブランチ、Push対象コミットを再確認する
3. 想定外の差分や認証上の問題があればPushせずに停止する
4. 現在のリモートブランチへPushする
5. `git status` とローカル・リモートのコミットIDを確認し、結果を日本語で報告する

## 禁止事項

- 「コミットして」「publishして」「公開して」だけをPush承認として扱わない
- 明示的な「Pushしてください」がない状態でPushしない
- 依頼対象外のファイルや未追跡ファイルをステージしない
- `.env*` や、APIキー・秘密鍵らしき値を含むファイルをステージしない
- 検証失敗、秘密情報の疑い、想定外の変更がある状態でコミットしない
- ユーザーの承認なしに `--no-verify` 等でチェックをスキップしない
