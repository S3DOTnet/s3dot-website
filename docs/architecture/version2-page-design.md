# S3DOT Version2 ページ設計書

## 文書情報

- 状態: 承認待ち（実装未着手）
- 対象: S3DOTコーポレートサイト Version2
- 目的: ページ構成、導線、CTA、SEO、実装優先順位の基準を定める
- 前提: 本設計書の承認後に実装へ進む。承認前はコード・設定を変更しない

## 1. Version2の目的

Version2では、S3DOTを「AIをもっと身近にする会社」として分かりやすく伝え、企業が安心して相談できるサイトを目指す。

最終目的は問い合わせ獲得とし、次の方針で設計する。

1. AIに詳しくない担当者にも、支援内容と相談後の流れが理解できるようにする
2. 会社情報、サービス、料金、事例、安全性に関する情報を整理し、信頼感を高める
3. 無料相談フォームまたは公式LINEへ、迷わず移動できる導線を作る
4. 建設業だけに特化せず、幅広い業種の中小企業に向けたAI導入支援として設計する
5. 高額なシステム開発だけでなく、小さく始められる現実的なAI活用を伝える

### 成果の考え方

Version2の成果は、単純な閲覧数だけではなく、次の行動が増えたかで判断する。

- 無料相談フォームへの到達と送信
- 公式LINE相談リンクのクリック
- サービス・料金・事例ページの閲覧
- 各ページから問い合わせページへの遷移

## 2. 最優先の対象顧客

### 主な対象

- AIを導入したいが、何から始めればよいか分からない中小企業
- 事務作業、問い合わせ対応、資料作成、集計などの定型業務を減らしたい企業
- 高額なシステム開発をいきなり行うのではなく、現実的な価格でAI導入を始めたい企業

### 想定する閲覧者

- 中小企業の経営者
- 業務改善を担当する管理職・担当者
- AI導入を任されたが、専門知識を持っていない担当者

### 閲覧者が感じやすい不安

- 自社で何にAIを使えるか分からない
- 導入費用や運用費用が分からない
- 専門用語が多く、相談してよい段階か判断できない
- 導入後に社内で使い続けられるか不安
- 情報管理や安全性が心配

各ページでは、この不安に対して「対象業務」「費用の考え方」「導入の流れ」「支援範囲」「相談方法」を具体的に示す。

## 3. 最終的に促したい行動

優先順位は次のとおりとする。

1. 無料相談フォームを送信する
2. 公式LINEで相談する
3. サービス詳細、活用イメージ・事例、料金を確認する

サイト内で複数の行動を同じ強さで並べすぎず、原則として「無料相談」を主要CTAにする。LINE相談は、フォーム入力の負担を感じる人向けの補助CTAとする。

### 導入事例の表記ルール

- 実在する顧客の事例は、顧客から掲載許可を得たものだけを「導入事例」と表記する
- 掲載許可がない内容、想定例、試作例は「活用イメージ」と明記する
- 実績があるように誤認させる見出し、数値、写真、顧客属性、説明は使用しない
- 匿名事例であっても、実在する顧客情報を基にする場合は、掲載範囲について許可を得る
- `/case` のナビゲーション表示名は「活用イメージ・事例」とする
- 公開前に、各掲載内容が「導入事例」「活用イメージ」のどちらに該当するかを個別に確認する

確認できていない項目を、既存ページの表記や一般的な会社情報から推測して掲載しない。複数のページに同じ会社情報を掲載する場合は、表記を統一する。

### 料金表記のルール

- 確定している料金だけを掲載する
- 案件内容によって変動する費用は「要相談」「個別見積もり」などと明記する
- 「格安」「高額」「最安」など、比較対象と根拠を示せない表現は使用しない
- 初期費用、追加費用、月額費用、外部サービス利用料が発生する場合は、利用者が誤解しない形で記載する
- 料金に含まれる作業と含まれない作業を可能な範囲で示す
- 公開前に、料金、税表記、契約条件、支払い条件の整合性を確認する

## 5. ヘッダーとフッターの正式なリンク設計

### 共通方針

- グローバルナビゲーションは独立したページURLに統一する
- `#service` と `/service` のような、トップページ内アンカーと下層ページURLの混在をなくす
- どのページから操作しても同じURLへ移動するようにする
- トップページ内アンカーは、ページ内の補助的な目次などに限定する
- ロゴは常にトップページ `/` へ移動する
- 現在地が分かる表示を設ける
- キーボード操作とスクリーンリーダーで利用できる構造にする

### PCヘッダー

左から右へ、次の順番を基本とする。

| 表示 | 遷移先 | 種別 |
| --- | --- | --- |
| S3DOTロゴ | `/` | ホームリンク |
| サービス | `/service` | 通常ナビゲーション |
| 活用イメージ・事例 | `/case` | 通常ナビゲーション |
| 料金 | `/price` | 通常ナビゲーション |
| FAQ | `/faq` | 通常ナビゲーション |
| 会社情報 | `/company` | 通常ナビゲーション |
| 無料相談 | `/contact#contact-form` | 主要CTA |

### スマートフォンヘッダー

- 常時表示する要素は、S3DOTロゴとメニューボタンに絞る
- メニューボタンには開閉状態が分かるラベルを付ける
- メニューを開いた後のリンク順は、PCと同じにする
- メニュー末尾に「無料相談」と「公式LINEで相談」を配置する
- リンク選択後はメニューを閉じる
- Escapeキーで閉じられるようにする
- メニュー表示中は背景へフォーカスが移動しないようにする

### フッター

フッターは次のグループに分ける。

| グループ | 表示と遷移先 |
| --- | --- |
| サービス情報 | サービス `/service`、活用イメージ・事例 `/case`、料金 `/price`、FAQ `/faq` |
| 会社・相談 | 会社情報 `/company`、無料相談・お問い合わせ `/contact`、公式LINE 外部URL |
| 法務・案内 | プライバシーポリシー `/privacy`、利用規約 `/terms`、特定商取引法に基づく表記 `/legal`、サイトマップ `/sitemap` |
| 会社連絡先 | 会社名、所在地、電話番号、メールアドレス |

ヘッダー・フッターでは、`#concept`、`#service`、`#story`、`#contact`、`#company`を正式リンクとして使用しない。

### 旧アンカーリンクの互換性方針

現行トップページでは、次のアンカーが実際に定義されていることを確認済みである。

- `#concept`
- `#service`
- `#story`
- `#contact`
- `#company`
- `#process`
- `#faq`
- `#contact-form`

また、現行のヘッダー、フッター、ページ内CTAでは、特に `#concept`、`#service`、`#story`、`#contact`、`#company` が参照されている。Version2で残すトップページ内セクションは未確定だが、公開後も外部サイトのリンクや既存ブックマークから適切な案内へ到達できるよう、次の方針とする。

1. Version2のトップページに対応する概要セクションが残る場合は、既存の `id` を維持する
2. 詳細情報は、各概要セクションから `/service`、`/company`、`/contact` などの正式ページへ案内する
3. セクションを統合・削除する場合は、同じ意味を持つ最も近い位置に互換用アンカーを残し、正式ページへの明確なリンクを置く
4. 既存アンカーを削除する前に、サイト内参照、sitemapの過去設定、共有済みURLを確認する
5. アンカーURLは互換性のために残しても、canonicalやsitemap.xmlの登録対象にはしない

想定する主な案内先は次のとおりとする。

| 旧アンカー | Version2での表示位置 | 詳細案内先 |
| --- | --- | --- |
| `#concept` | トップページの理念・価値提案 | `/company` またはトップページ内の対応セクション |
| `#service` | トップページのサービス概要 | `/service` |
| `#story` | トップページの理念・ストーリー概要 | `/company` |
| `#contact` | トップページ下部の相談CTA | `/contact` |
| `#company` | トップページの会社概要 | `/company` |
| `#process` | トップページの導入の流れ概要 | `/service` |
| `#faq` | トップページのFAQ概要 | `/faq` |
| `#contact-form` | 問い合わせページのフォーム | `/contact#contact-form` |

第1段階の実装前に、公開環境で使われているアンカーを再確認し、この表に不足がないことを確認する。

## 6. CTA設計

### CTAの使い分け

| CTA | 目的 | 遷移先 | 表示上の優先度 |
| --- | --- | --- | --- |
| 無料相談 | 具体的な課題や導入相談をフォームで受け付ける | `/contact#contact-form` | 最優先。強調色の主要ボタン |
| 公式LINEで相談 | 短い質問や、フォーム入力前の気軽な相談を受け付ける | 公式LINEの外部URL | 2番目。LINEと分かる色・アイコン |
| お問い合わせ | 取材、協業、契約、その他の一般的な連絡を受け付ける | `/contact` | 通常リンクまたは補助ボタン |
| サービスを見る | 支援範囲を確認してもらう | `/service` | 情報確認用 |
| 料金を見る | 費用の考え方を確認してもらう | `/price` | 情報確認用 |

「無料相談」と「お問い合わせ」は遷移先が近いため、ボタン文言で目的を区別する。問い合わせページでは、無料相談フォームと一般的な連絡方法を同じページ内で明確に分ける。

### ページごとの主要CTA

| ページ | 主要CTA | 補助CTA | ページ下部CTA |
| --- | --- | --- | --- |
| トップ | 無料相談 | 公式LINE、サービスを見る | 無料相談＋公式LINE |
| サービス | 無料相談 | 料金を見る、活用イメージ・事例を見る | サービス内容を踏まえた無料相談 |
| 活用イメージ・事例 | 無料相談 | サービスを見る | 同じ課題について無料相談 |
| 料金 | 無料相談 | 公式LINE | 自社の場合の費用を無料相談 |
| FAQ | 無料相談 | 公式LINE | 解決しない疑問を相談 |
| お問い合わせ | フォームを送信 | 公式LINE、電話、メール | 原則としてフォーム本体を最終CTAとする |
| 会社情報 | 無料相談 | サービスを見る | S3DOTへ相談する |
| 法務ページ | お問い合わせ | なし | 強い営業CTAは置かず、フッター導線を基本とする |

### CTA配置ルール

- 各ページのファーストビューには、主要CTAを原則1つ置く
- LINEを併記する場合も、無料相談より強く見せない
- 長いページでは、内容の区切りとページ下部にCTAを再掲する
- ボタン文言は「詳しくはこちら」ではなく、移動後の行動が分かる表現にする
- 外部リンクは外部サイトへ移動することが分かるようにする
- 問い合わせの返信目安は「原則2営業日以内」に統一する
- 問い合わせページ、フォーム付近の案内、自動返信メールで異なる返信目安を表示しない

## 7. SEO設計

### 共通方針

- 各ページに固有の `title`、`description`、`canonical`、`h1` を設定する
- canonicalは、そのページ自身の正規URLを絶対URLで指定する
- 本番URLは正式Webサイトの `https://s3dot.com` に統一し、`www`付きホストなどからは正規ホストへ転送する
- `h1`は各ページに1つだけ置き、そのページの主題を表す
- トップページ内アンカーはsitemap.xmlへ登録しない
- sitemap.xmlには、検索結果として表示させたい独立ページだけを登録する
- API、404ページ、管理用ページ、重複ページはsitemap.xmlへ登録しない
- 実際に提供していないサービスや、掲載条件を満たしていない実績をSEO文言に含めない

### SEO文言の確認ルール

- 次の表に記載するtitleとdescriptionは、現時点では確定文言ではなく設計案として扱う
- 実装時に、実際のページ内容との一致、ページ間の重複、検索結果での表示を考慮した文字数を再確認する
- ページ本文にないサービス、対応範囲、実績、料金をtitleやdescriptionだけに追加しない
- 公開可能な事例がない段階では、「導入事例」を中心としたSEO文言にしない
- 文言を確定する際は、各ページのh1と役割も同時に確認する

## 9. 実装順序

実装は、次の5段階に分け、順番を変更せずに進める。

- 1段階ごとに変更対象、予定する変更、検証方法を提示し、承認を得てから着手する
- 着手した段階の変更だけを行い、後の段階の変更を先に混在させない
- 各段階の検証と報告が完了してから、次の段階へ進む
- 想定外の変更、秘密情報の疑い、仕様判断が発生した場合は、その段階の作業を停止する

### 第1段階：ヘッダー、フッター、ページ導線、`/company`

- 正式なページURLに統一する
- `/company`を用意する
- PC・スマートフォンのナビゲーションをそろえる
- 無料相談、LINE、サービス、料金への遷移先を統一する
- 旧アンカーの互換性を維持し、正式ページへの案内を追加する
- 会社情報は正式情報との照合が完了した項目だけを掲載する

#### 変更予定のファイル候補

#### 第1段階の検証方法

1. `git status --short`と対象ファイルの差分で、第1段階以外の変更がないことを確認する
2. `git diff --check`で空白・形式上の問題がないことを確認する
3. `npm run lint`と`npm run build`が成功することを確認する
4. Next.js DevToolsで`/company`を含むルート一覧と実行時エラーを確認する
5. トップ、サービス、活用イメージ・事例、料金、FAQ、お問い合わせ、会社情報、法務ページから、ヘッダーとフッターの全リンクを確認する
6. PC表示とスマートフォン表示で、メニューの開閉、リンク遷移、無料相談、LINE相談を確認する
7. `/#concept`、`/#service`、`/#story`、`/#contact`、`/#company`、`/#process`、`/#faq`が互換用の案内位置へ移動できることを確認する
8. `/contact#contact-form`が問い合わせフォームへ移動できることを確認する
9. 主要ページで404、リンク切れ、画面上のはみ出し、実行時エラーがないことを確認する
10. Playwright MCPは使用せず、第1段階の実装・検証完了後に導入要否を検討する

第1段階だけを本番公開するかは別途確認する。第2段階のSEO修正前に新規`/company`を公開する場合、metadataとcanonicalが未修正の状態にならないかを公開前に確認する。

### 第2段階：metadata、canonical、sitemap、h1

- ページ固有のtitleとdescriptionを設定する
- 各ページ自身をcanonicalにする
- sitemap.xmlからトップページ内アンカーを除外する
- 独立ページをsitemap.xmlへ登録する
- 各ページのh1を1つにする
- SEO文言がページ内容と一致し、重複や不適切な実績表現がないことを確認する

### 第3段階：問い合わせAPIの安全性

- メール本文へ入れる入力値を安全に処理する
- ハニーポットの処理順を見直す
- サーバーレス環境で有効なレート制限を検討する
- 利用者へ内部設定や外部サービスの詳細を返しすぎない
- 正常送信、入力エラー、送信失敗をテストする

### 第4段階：アクセシビリティと表示速度

- メニューのフォーカス管理とキーボード操作を改善する
- `prefers-reduced-motion`へ対応する
- 文字サイズ、色のコントラスト、フォーカス表示を確認する
- ロゴ画像を最適化し、ブラウザ上の画像加工を見直す
- 不要なクライアントコンポーネントとアニメーションを減らす

### 第5段階：デザイン再設計

- 承認済みのページ構成に沿ってワイヤーフレームを作る
- 共通の文字、色、余白、ボタン、カードのルールを定める
- トップページから再設計し、その後に下層ページへ展開する
- 最終的にPC・スマートフォンの全主要ページを確認する

各段階は、対象範囲、変更内容、検証方法について承認を得てから実装する。後の段階の変更を先行して混在させない。

## 10. 完了条件

Version2は、次の条件をすべて満たした時点で完了とする。

### ページ・導線

- 全ページのヘッダーとフッターから主要ページへ正しく移動できる
- トップページ以外でもリンク切れや存在しないアンカーへの移動がない
- 無料相談、公式LINE、サービス、料金の遷移先が設計どおりである
- PC・スマートフォンでナビゲーションとCTAが画面外へはみ出さず、操作できる

### SEO

- 公開する各ページに固有のtitleとdescriptionがある
- 各ページのcanonicalが自身の正規URLを指している
- 各ページのh1が1つで、ページ内容と一致している
- sitemap.xmlに公開対象の独立ページが登録されている
- sitemap.xmlにトップページ内アンカー、API、404ページが登録されていない
- titleとdescriptionが実際の提供内容・公開可能な実績と一致している

### 表記・公開情報

- 導入事例は顧客から掲載許可を得た内容だけである
- 想定例、試作例、掲載許可のない内容は「活用イメージ」と明記されている
- 会社情報が正式情報と照合され、未確認項目を推測で掲載していない
- 料金、追加費用、月額費用、外部サービス利用料が誤解のない形で記載されている
- 根拠のない価格比較表現を使用していない

### 問い合わせ

- 入力値の検証がサーバー側で行われる
- メール本文へ入力値を安全に反映できる
- 迷惑送信への対策がある
- 内部設定や秘密情報をエラーメッセージで公開しない
- 送信成功、入力エラー、送信失敗時に適切な案内が表示される

### アクセシビリティ・表示速度

- メニューとフォームをキーボードで操作できる
- フォーカス位置が視覚的に分かる
- モーション軽減設定を尊重する
- 主要コンテンツとロゴが不必要に遅れて表示されない
- PC・スマートフォンで主要導線が破綻しない

### 検証

- lintとbuildが成功する
- 主要ページで実行時エラーがない
- PC・スマートフォンで主要導線を確認する
- 問い合わせフォームの正常系と異常系を確認する
- 意図しないファイル変更、秘密情報、個人情報が含まれていない
