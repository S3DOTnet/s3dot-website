# S3DOT 開発ルール

このファイルは、S3DOTウェブサイトの開発に関わる人・AIエージェントの双方が参照できるよう、既存ルールの要点を整理したものです。詳細な指示・最新の正本は各参照先ファイルにあります。

## ルールの優先順位

1. [CLAUDE.md](./CLAUDE.md) / [AGENTS.md](./AGENTS.md) — Claude Code / Codexなどエージェントへの指示。最も詳細かつ優先
2. このDEVELOPMENT_RULES.md — 人・エージェント双方向けの要点整理
3. [docs/README.md](./docs/README.md) — 開発記録の運用ルール

内容に矛盾がある場合は、CLAUDE.md / AGENTS.mdを優先してください。

## 開発方針

S3DOTは短期的なサイト制作ではなく、AI導入・業務改善サービスへ成長する企業である。個々の変更を判断する際も、この方向性を常に考慮する。

## 基本ルール

既存サイトのデザイン・構成・機能を尊重する。

変更する場合は、必ず事前に次の4点を説明してから実行する。

1. 変更理由
2. 変更対象
3. メリット
4. 影響範囲

## 禁止事項

以下は、確認なしで行わない。

- ページ削除
- 大幅なデザイン変更
- レイアウト変更
- 不要なライブラリ追加
- 既存機能の削除
- フォーム動作変更

## 作業の基本方針

- 依頼対象外の既存コード・設定は変更しない
- 変更前に、目的と変更対象を日本語で簡潔に説明する(「基本ルール」の4点を含める)
- 仕様・デザイン・公開範囲など、途中で判断が必要になった場合は作業を停止してユーザーへ確認する
- 変更後は `npm run lint` と `npm run build` を実行し、両方の成功を確認する(必要に応じてPC/スマホ双方の表示確認も行う)
- 検証成功後は簡潔な日本語のコミットメッセージでコミットする。pushはユーザーが明示的に「Pushしてください」と指示するまで行わない
- lint/buildの失敗、機密情報の混入、意図しない差分を発見した場合は、コミット・pushせず停止して報告する

## ブランドガイドライン

- S3DOTのブランドは高級感・未来感・信頼感を最優先する
- 配色は黒基調に、青・紫系のアクセントを使用する
- 公式ロゴ(`public/images/logo.png`、`src/components/ui/TransparentLogo.tsx`)の比率・形状・発光表現を無断で変更しない
- PC表示だけでなく、必ずスマホ表示も確認する

## 品質基準

- SEO、アクセシビリティ、表示速度を常に意識する
- 不要なパッケージを追加しない

## セキュリティ

- 環境変数・APIキー・認証情報・秘密鍵の値を出力・ログ・コミットに含めない
- `.env.local` など `.env*` ファイルはコミットしない
- GmailおよびGoogle Driveの検索・閲覧・使用は、ユーザーから明示的に依頼された場合に限る

## 表記ルール(Version2設計書より抜粋)

詳細は [docs/architecture/version2-page-design.md](./docs/architecture/version2-page-design.md) を参照。

- 「導入事例」は、顧客から掲載許可を得た実績にのみ使用する
- 「活用イメージ」は、想定例・試作例・掲載許可のない内容に使用する
- 料金は確定分のみ掲載し、変動費用は「要相談」「個別見積もり」と明記する
- 問い合わせ返信目安は「原則2営業日以内」で全ページ・自動返信メールを統一する
- 未確定情報(設立年月、事業内容の正式文言など)は推測で補わず、確定後に掲載する

## 開発記録

- 日報: `docs/daily/YYYY-MM-DD.md`(`/daily-report` Skillを使用可)
- 週報・月報・会議・アイデア・不具合・変更履歴・設計・プロンプト・知識メモ: `docs/` 配下の各フォルダ(分類は[docs/README.md](./docs/README.md)参照)
- 確認できた事実と、提案・推測を区別して記載する
- APIキー、認証トークン、秘密鍵、個人情報を記録しない
- 同じ日の日報が存在する場合は、そのファイルを更新し重複作成しない
- 手書きの所感やメモは削除せずに保持する
